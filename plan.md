## Interview: Verticale Eventi Volontiamo

Implementare una nuova verticale Eventi end-to-end: menu laterale web, lista tabellare paginata, filtro per nome e stato, creazione evento, cancellazione logica, API REST sottile, dominio testato L0, persistenza PostgreSQL via EF Core e test L1. Approccio raccomandato: replicare e migliorare il pattern Users, mantenendo la logica nel dominio, usando ID evento numerico generato dal database, note operative in Markdown persistito come testo, e rendering rich text in anteprima tabellare con react-markdown senza HTML raw.

**Steps**
1. Preparazione e contratti di dominio. Definire `EventStatus` con `Draft = 0`, `Active = 1`, `Concluded = 2`; creare `Event`, `CreateEventRequest`, `EventResponse`, `EventListFilter`, `IEventRepository`, `EventService`. *blocca 2, 3, 4, 5*
2. TDD L0 dominio. Scrivere prima `EventServiceTests` con fake repository in memoria: validazione nome obbligatorio, start/end UTC coerenti, location opzionale, note Markdown accettate come testo lungo, default filtro active+draft, filtro nome case-insensitive delegato al repository, paginazione normalizzata, delete soft. *depends on 1 shape; guida 3*
3. Implementazione dominio. Implementare `Event.Create`, `Event.SoftDelete`, validazioni e mapping response nel servizio fino a far passare L0. *depends on 2*
4. Persistenza EF Core. Aggiungere DbSet Events e configurazione in `AppDbContext`: tabella `events`, PK `id` integer generated by identity, colonne snake_case, `name` required max length, `start_at_utc`, `end_at_utc`, `location` nullable, `operational_notes_markdown` text, `status` int, audit, `is_deleted`, query filter, indici su name/status/start. Creare `EventRepository` con filtro nome/status/paginazione ordinata per start asc then name. *depends on 3; parallel with API endpoint skeleton after 1*
5. Migrazione database. Generare migration EF `AddEvents` e verificare schema PostgreSQL, inclusa identity integer e colonna text per note. *depends on 4*
6. API Events. Creare `Events/Contracts.cs` global aliases se utili e `EventEndpoints` con `POST /api/v1/events`, `GET /api/v1/events`, `DELETE /api/v1/events/{id:int}`. Registrare `IEventRepository`, `EventService` e `MapEventEndpoints` in `Program.cs`. API resta sottile e traduce `Result<T>` in HTTP come Users. *depends on 3; completa dopo 4*
7. TDD L1 API. Aggiungere `EventsEndpointTests` con Testcontainers esistente: create valid returns 201 with numeric id, invalid dates returns validation problem, list default excludes concluded and includes active/draft, filter by name works, status filter can include concluded/all, pagination works, delete returns 204 and excludes from list/get/list. *depends on 5, 6*
8. Web shell extraction. Prima di aggiungere Eventi, estrarre la shell duplicata da home/users in componenti condivisi o layout applicativo leggero: header, sidebar/mobile nav, nav items Home/Volontari/Eventi, stato attivo per route. *parallel with 9 after API contract is known*
9. Web adapter eventi. Aggiungere `lib/events/contracts.ts` e `http-events-adapter.ts` con funzioni server-only: read events page, create event, delete event, validazione payload/errori come `http-users-adapter`, query `name`, `status`, `page`, `pageSize`. *depends on API contract step 6*
10. Web route lista eventi. Creare `app/events/page.tsx` server component: default query statuses active+draft, form GET per filtro nome/stato, tabella paginata, anteprima Markdown troncata con `react-markdown`, azione delete con conferma minima e revalidate/redirect. *depends on 8, 9*
11. Web route creazione. Creare `app/events/new/page.tsx` con form nome, date/ore locali Europe/Rome, location opzionale, note Markdown textarea ampia, stato selezionabile con default bozza, submit server action verso API e redirect alla lista. *depends on 9; parallel with 10 after 8*
12. Dipendenza Markdown e docs Next. Aggiungere `react-markdown` a package.json/package-lock; prima dell'implementazione effettiva, installare dipendenze e verificare se i docs locali Next 16 su forms/server actions sono disponibili in node_modules. *depends on 10/11 design; blocks final web implementation details*
13. Rifinitura UX e accessibilita. Applicare stile coerente con la shell LILT: sidebar desktop e drawer mobile, tabella leggibile responsive, bottoni con stati focus, empty/error states, copy operativo in italiano, nessuna card annidata. *depends on 10, 11*
14. Verifica completa. Eseguire L0, poi L1, poi lint/build frontend. Correggere solo regressioni legate agli eventi. *depends on all implementation steps*

**Decisions**
- Inclusi: evento con `int Id`, nome, start/end date-time, location opzionale, note operative Markdown, stato, lista tabellare, filtro nome, filtro stato, default active+draft, creazione web, cancellazione logica, API POST/GET/DELETE, EF migration, L0 dominio, L1 API, voce menu laterale/mobile.
- Esclusi: modifica evento post-creazione, dettaglio evento read-only, campi indirizzo strutturati, unicita del nome, test automatici frontend/E2E, autenticazione/autorizzazione, import/export, assegnazione volontari agli eventi.
- Date/ore: la UI lavora in ora locale Italia Europe/Rome; API/domain persistono istanti UTC e validano `EndAtUtc >= StartAtUtc`.
- Stato: form di creazione con stato selezionabile e default bozza; la lista default mostra active+draft, con filtro per active, draft, concluded o tutti.
- Note: salvate come Markdown testuale in colonna `text`; rendering web con `react-markdown`, senza HTML raw.
- Cancellazione: soft delete con `is_deleted` e query filter EF come Users.
- Architettura: API resta adapter REST; business logic e validazione in `volontiamo.domain`; repository e DB in `volontiamo.api.Persistence`; test L0 usano fake in memoria dedicato.
